ℹ️ Acerca de este proyecto

Documentación sobre la metodología, las fuentes de datos y la estructura del catálogo de Open Data Canarias.

🔬 Metodología de recopilación

Los datos se recopilan manualmente a partir de fuentes oficiales: portales de transparencia, portales CKAN, sedes electrónicas y páginas institucionales de cada entidad. Cada entrada se valida contra el schema JSON oficial del proyecto antes de incorporarse al catálogo.

El campo verification.status indica el nivel de confianza de cada registro:

  • verified — dato comprobado directamente en fuente oficial.
  • estimated — estimado a partir de indicios fiables.
  • inferred — deducido de otras fuentes.
  • pending — pendiente de verificación.

🏛️ Tipos de entidad

El campo entity_kind clasifica cada institución según su naturaleza administrativa. La distinción es relevante para entender el alcance y la obligatoriedad de publicación de datos.

Tipo Descripción Ejemplos
gobierno_autonomico Gobierno de la Comunidad Autónoma de Canarias y sus consejerías Gobierno de Canarias, datos.canarias.es
cabildo Corporaciones insulares, una por cada isla habitada Cabildo de Tenerife, Cabildo de Gran Canaria
ayuntamiento Administración local municipal Ayuntamiento de Las Palmas de Gran Canaria, Santa Cruz de Tenerife
organismo Entidades especializadas con misión sectorial: estadística, cartografía, universidades, puertos… ISTAC, GRAFCAN, Universidad de La Laguna
empresa_publica Sociedades mercantiles de capital público PROEXCA, Turismo de Canarias

📈 Criterios de madurez de portal

Cada portal recibe automáticamente un nivel de madurez calculado a partir de cuatro factores objetivos: formatos publicados, disponibilidad de API, tecnología del portal y volumen de datasets. El objetivo es reflejar de forma homogénea qué tan reutilizables son los datos publicados.

Nivel Criterios (todos deben cumplirse) Interpretación
🟢 Alta Formatos legibles por máquina (CSV, JSON, GeoJSON…) + API declarada + tecnología robusta (CKAN, ArcGIS, eDAtos, IDE…) + ≥ 50 datasets Ecosistema avanzado, datos directamente consumibles por aplicaciones
🟠 Media Formatos legibles por máquina, pero sin API declarada, tecnología básica o volumen de datasets reducido Datos accesibles pero con limitaciones para la reutilización automatizada
🔴 Baja Sin formatos legibles por máquina (solo PDFs o HTML), o portal de transparencia estático sin API ni datasets estructurados Información disponible para consulta humana, difícil de reutilizar

Los formatos considerados legibles por máquina son: tabulares (CSV, JSON, XML, XLSX, XLS, ODS, TSV), geoespaciales (GeoJSON, KML, SHP, GML, GPKG, GDB), linked data (RDF, RDF-XML, N3, Turtle), estadísticos y transporte (PC-Axis, GTFS) y científicos (NetCDF). Los formatos PDF y HTML no puntúan como machine-readable a efectos de este cálculo.

🧩 Campos principales del catálogo

Campo Descripción Valores posibles
id Identificador único de la entidad Texto en minúsculas con guiones bajos
name Nombre oficial de la entidad Texto libre
entity_kind Tipo de entidad gobierno_autonomico · cabildo · ayuntamiento · organismo · empresa_publica
scope Ámbito administrativo autonomico · insular · municipal · estatal · europeo
islands Islas a las que pertenece la entidad Tenerife · Gran Canaria · Lanzarote · Fuerteventura · La Palma · La Gomera · El Hierro · La Graciosa · Todas
portals Lista de portales publicados por la entidad Array de objetos (ver schema)
portals.kind Tipo de portal open_data · transparencia · estadistica · geoespacial · agregador · other
portals.machine_readable ¿Los datos son legibles por máquina? true / false
portals.has_api ¿Dispone de API? true / false / null
portals.dataset_count Número aproximado de datasets publicados Entero o null si no está disponible
portals.last_updated Fecha de última revisión del portal Formato YYYY-MM
verification.status Nivel de confianza del registro verified · estimated · inferred · pending

🗓️ Actualización del catálogo

El catálogo se actualiza manualmente. La fecha de última revisión de cada portal se refleja en el campo portals.last_updated (formato YYYY-MM). El script src/static/data/fecha.py automatiza la consulta a APIs CKAN para mantener esas fechas al día.
